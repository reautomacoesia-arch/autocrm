## 1. Propostas — Editar proposta

**Arquivo:** `components/proposals/ProposalDetail.tsx`

- Botão "✏️ Editar" no cabeçalho do `ProposalDetail`
- Ao clicar, ativa modo de edição inline com campos: valor (`value: number`), validade (`valid_until: string | null`), observações (`notes: string | null`)
- Botões Salvar / Cancelar
- Salvar chama `PATCH /api/proposals/{id}` com allowlist explícita: `value`, `valid_until`, `notes`
- Após salvar, atualiza o estado local (`setProposal`) e sai do modo de edição
- Itens da proposta (`proposal_items`) permanecem somente leitura

**API:** `app/api/proposals/[id]/route.ts`
- Verificar se PATCH existe e cobre os campos; se usar spread (`{ ...body }`), corrigir para allowlist

## 2. Financeiro Global — CRUD de Transações

**Arquivos:**
- `app/(dashboard)/financial/page.tsx` — extrair lista de transações para Client Component
- `components/financial/TransactionManager.tsx` — novo Client Component

**Comportamento:**
- `TransactionManager` recebe `initialTransactions` e `clients` como props do Server Component pai
- Botão "Registrar transação" abre formulário inline (não modal) com campos: cliente (select), valor, tipo (recebido/pendente), data, descrição
- Cada transação na lista: clique → edit inline com os mesmos campos; badge de status visível
- Ícone Trash2 → `window.confirm` + `DELETE /api/transactions/{id}` + remoção otimista
- Totais (MRR mantido no Server Component; total recebido e pendente recalculados no estado local)
- Botão "Registrar" chama `POST /api/transactions`; após sucesso, adiciona ao estado local
- PATCH de edição: `PATCH /api/transactions/{id}` com allowlist: `amount`, `type`, `date`, `description`
- Badge "Atrasado" (vermelho) para transações `pending` cuja `date < hoje` (parseDate sem UTC shift)

## 3. Tarefas — Detalhes inline + status bidirecional

**Arquivo:** `components/tasks/TaskList.tsx`

### 3a. Status bidirecional
- `STATUS_NEXT` atual: `pending → in_progress → done → null` (sem retorno)
- Nova lógica: clicar no círculo verde ("done") reseta para `pending`
- Logicamente: `done` → ao clicar → `pending` (toggle off)
- Visual: círculo verde mostra ✕ ao hover (via CSS group-hover ou title="Reabrir")
- PATCH via `api/tasks/{id}` com `{ status: 'pending' }`

### 3b. Expansão de detalhes inline
- Clicar no **título** da tarefa expande/colapsa uma seção accordion no mesmo card
- Seção expandida mostra: descrição completa, cliente vinculado (link para `/clients/{id}`), data de vencimento formatada, prioridade
- Estado: `expandedId: string | null` no componente
- `stopPropagation` no botão circular e no Trash2 para não acionar expand

## 4. Dashboard — Calendário Mensal

**Arquivos:**
- `app/api/calendar/route.ts` — nova rota GET
- `components/dashboard/DashboardCalendar.tsx` — novo Client Component
- `app/(dashboard)/page.tsx` — inserir `<DashboardCalendar />` após os metric cards

### API: `GET /api/calendar?year=YYYY&month=MM`
Retorna:
```json
{
  "tasks": [{ "id", "title", "due_date", "status" }],
  "transactions": [{ "id", "amount", "date", "type", "clients": { "name" } }]
}
```
- `tasks`: `due_date` dentro do mês, `status != 'done'`
- `transactions`: `date` dentro do mês, `type = 'pending'`
- Filtro de data: `>= YYYY-MM-01` e `<= YYYY-MM-{lastDay}`

### Componente `DashboardCalendar`
- Estado: `year` e `month` (inicializados com mês/ano atual)
- Ao montar e a cada mudança de `year`/`month`: `fetch('/api/calendar?year=&month=')`
- Cabeçalho: `← Maio 2026 →` (botões de navegação)
- Grid: 7 colunas (Dom–Sab), dias do mês com cells vazias para alinhar o primeiro dia
- Cada cell de dia:
  - Número do dia (destacado se hoje: borda indigo)
  - Chips compactos (máx 2 por categoria):
    - Indigo com texto truncado: tarefas com `due_date` naquele dia
    - Âmbar com valor formatado: transações pendentes com `date` naquele dia
    - "+N" cinza se houver mais itens além do limite
  - `title` no chip com o nome/valor completo (tooltip nativo)

**Arquivos:**

- Layout:
  - Título "Histórico de Atividades"
  - Lista de interações: ícone tipo | descrição | nome do cliente (link `/clients/{id}`) | data/hora
  - Paginação no rodapé: `← Anterior | Página X de Y | Próxima →`
  - Links de paginação: `href="/activity?page=N"`

## Regras Técnicas

- Next.js App Router: `params` e `searchParams` são `Promise` — sempre `await`
- Nunca `{ ...body }` em PATCH — allowlist explícita
- Datas `YYYY-MM-DD` parseadas com `new Date(year, month-1, day)` para evitar UTC shift
- Server Components não recebem funções como props de clientes
- Supabase com RLS — todas as rotas usam `createClient()` do server
